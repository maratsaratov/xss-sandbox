# XSS-песочница — обучающий стенд

Интерактивный стенд для наглядной демонстрации и исправления уязвимости
**межсайтового скриптинга (XSS)**. По одной странице на каждый подтип:

1. **Хранимый (постоянный) XSS** — `/stored`
2. **Отражённый (непостоянный) XSS** — `/reflected`
3. **DOM-модель (XSS на стороне клиента)** — `/dom`

На каждой странице есть переключатель **«Уязвимый ⇄ Безопасный режим»** и
5 этапов: *где уязвимость и уязвимый код → практическая атака → как это видит
жертва → исправление кода → безопасный вид*.

> ⚠️ Стенд предназначен **только для обучения** на тренировочных данных в
> изолированной среде. Применение этих приёмов к чужим системам без разрешения
> противоправно.

## Технологии

| Слой        | Технология                | Почему                                            |
|-------------|---------------------------|---------------------------------------------------|
| Backend     | **Python + Flask**        | Минимум кода, легко читать на занятии             |
| База данных | **SQLite**                | Не нужен отдельный сервер БД — файл рядом с кодом  |
| Frontend    | **React (Vite)**          | Современный SPA; показывает XSS через `dangerouslySetInnerHTML` |
| Маршруты    | **react-router-dom**      | Реальные URL нужны для Reflected (`?q=`) и DOM (`#`) |

## Запуск через Docker

Нужен установленный **Docker Desktop** (с Docker Compose v2). Из корневой папки
проекта выполните:

```powershell
docker compose up --build
```

Поднимутся два контейнера:

| Сервис     | Контейнер      | Адрес                   |
|------------|----------------|-------------------------|
| `frontend` | `xss-frontend` | http://localhost:5173   |
| `backend`  | `xss-backend`  | http://localhost:5000   |

Когда в логах появятся строки `Running on http://...:5000` и
`Local: http://localhost:5173/`, откройте в браузере:

```
http://localhost:5173
```

Первая сборка занимает несколько минут (скачивание образов и установка
зависимостей). Запросы `/api/*` фронтенд проксирует на сервис `backend`
по внутренней сети Docker.

Остановить: `Ctrl+C`, затем при желании `docker compose down`.

## Где в коде отмечена уязвимость (для методички)

Все ключевые места помечены комментариями `УЯЗВИМОСТЬ` и `ИСПРАВЛЕНО`.

| Подтип       | Уязвимый код                                                    | Исправление                                                  |
|--------------|-----------------------------------------------------------------|--------------------------------------------------------------|
| **Stored**   | `frontend/src/pages/StoredXSS.jsx` → `dangerouslySetInnerHTML`   | там же → вывод `{comment.text}` как текст                    |
|              | `backend/app.py` → `add_comment` (payload сохраняется в БД)     | `bleach.clean()` / `html.escape()` (доп. серверный рубеж)    |
| **Reflected**| `backend/app.py` → `reflected_search`: `echo = query`           | `backend/app.py`: `echo = html.escape(query)`               |
|              | `frontend/src/pages/ReflectedXSS.jsx` → `dangerouslySetInnerHTML`| вывод `{data.query}` как текст                              |
| **DOM**      | `frontend/src/pages/DomXSS.jsx` → `el.innerHTML = ...`           | `frontend/src/pages/DomXSS.jsx` → `el.textContent = ...`     |

## Сценарий демонстрации на занятии

Для каждой страницы:

1. **Шаг 1.** Показать уязвимый код (красная подсветка строк) — где именно
   данные пользователя попадают в HTML.
2. **Шаг 2.** Нажать «Подставить» на готовом payload и отправить его
   (комментарий / поиск / имя в `#`).
3. **Шаг 3.** В **уязвимом режиме** payload срабатывает: всплывает `alert`,
   а cookie сессии «утекает» в «Журнал сервера атакующего».
4. **Шаг 4.** Показать исправленный код (зелёная подсветка).
5. **Шаг 5.** Переключить тумблер в **🛡️ Безопасный режим** — тот же payload
   отображается как обычный текст и не выполняется.

### Что показывает «кража cookie»

При загрузке стенд ставит демонстрационную cookie
`session_token=SECRET-...` **без флага HttpOnly** — поэтому она доступна из
JavaScript. Payload `...onerror="window.__attacker('cookie='+document.cookie)"`
отправляет её в «Журнал сервера атакующего» (симуляция, наружу ничего не
уходит). Это показывает реальное последствие XSS — **угон сессии**.

> Защита от угона: ставьте cookie с флагами **HttpOnly** и **Secure** —
> тогда даже при XSS их нельзя прочитать из JS.

## Полезные приёмы во время демонстрации

- Кнопка **«Сбросить к исходным данным»** на странице Stored очищает
  внедрённые комментарии.
- Payload `<script>alert(1)</script>` **не сработает** через `innerHTML` —
  это специально показано, чтобы объяснить, почему атакующие используют
  `onerror` / `onload`.
- На страницах Reflected и DOM есть кнопка **«Скопировать вредоносную ссылку»** —
  можно показать, как именно ссылка доставляется жертве.

## Сводка способов защиты

- **Кодирование вывода (output encoding):** выводить данные как текст, а не как
  HTML. В React — без `dangerouslySetInnerHTML`; на сервере — `html.escape()`.
- **Санитизация HTML:** если разметку нужно разрешить — `DOMPurify` (клиент),
  `bleach` / `nh3` (сервер).
- **`textContent` вместо `innerHTML`** в нативном DOM-коде.
- **Флаги cookie:** `HttpOnly`, `Secure`, `SameSite`.
- **Content-Security-Policy (CSP):** запрещает выполнение постороннего скрипта
  даже при наличии бреши.
